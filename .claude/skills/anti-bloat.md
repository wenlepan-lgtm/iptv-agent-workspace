# anti-bloat-skill

防止 App 继续臃肿，控制代码复杂度。

## 触发方式
```
/anti-bloat
```

## 适用场景
- 新增功能前
- 代码审查时
- 重构建议时

## 检查规则

### 1. 抽象层检查
- ❌ 不新增 manager，除非证明必要
- ❌ 不新增 wrapper，除非证明必要
- ❌ 不新增 helper，除非证明必要
- ❌ 不新增 util，除非证明必要
- ✅ 新增抽象必须有至少 3 处真实复用

### 2. 依赖检查
- ❌ 不引入新库，除非现有方案不可行
- ✅ 必须对比现有方案和新库方案
- ✅ 必须评估新库的维护成本

### 3. 架构检查
- ❌ 不允许"小问题大改架构"
- ✅ 方案必须同时给出"重方案 vs 最小方案"
- ✅ 默认选最小方案

### 4. 代码量检查
- ⚠️ 单次新增超过 200 行需要审查
- ⚠️ 新增超过 5 个文件需要审查
- ⚠️ 新增超过 3 个类需要审查

## 输出模板

```markdown

### 变更概述
[描述变更内容]

### 检查结果

| 新增项 | 类型 | 是否必要 | 复用点 |
|--------|------|---------|--------|
| xxx | manager | 是/否 | [列出复用位置] |

#### 依赖
| 新增依赖 | 是否必要 | 替代方案 |
|---------|---------|---------|
| xxx | 是/否 | yyy |

#### 方案对比
| 方案 | 代码量 | 复杂度 | 推荐度 |
|------|--------|--------|--------|

| 完整方案 | 200行 | 中 | ⭐ |

### 建议
[基于检查结果的建议]

### 瘦身建议
1. [建议1]
2. [建议2]
```

## 决策树

```
需要新增抽象？
    │
    ├─ 是否有 3+ 处复用？
    │       ├─ 是 → 可以新增
    │       └─ 否 → 不新增，内联实现
    │
需要引入新依赖？
    │
    ├─ 现有方案是否可行？
    │       ├─ 是 → 使用现有方案
    │       └─ 否 → 评估新依赖
    │
需要改架构？
    │
    ├─ 是否是小问题？
    │       ├─ 是 → 最小改动
    │       └─ 否 → 架构审查
```

## 强制约束
- ✅ 必须证明新增抽象的必要性
- ✅ 必须对比最小方案和完整方案
- ❌ 不允许为了"优雅"而过度设计
