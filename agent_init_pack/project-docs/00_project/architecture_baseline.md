# architecture_baseline.md

## 1. 目标
定义本仓库的工程协作架构基线，而非具体业务系统代码架构。

## 2. 架构核心
本仓库采用 **Repo-Driven 协作架构**：

- 任务通过任务卡定义
- 开发通过开发记录落盘
- 审核通过审核记录落盘
- 验证通过验证记录落盘
- 全局状态通过索引文件总览

## 3. 模块划分

### 3.1 项目基线区
路径：`project-docs/00_project`

用途：
- 定义长期稳定信息
- 定义架构约束
- 定义验收口径
- 定义已知问题

### 3.2 任务区
路径：`project-docs/01_tasks`

用途：
- 一任务一卡
- 定义目标、约束、范围、验收标准

### 3.3 开发记录区
路径：`project-docs/02_dev_logs`

用途：
- 记录 Claude 的实际开发行为
- 记录修改范围与验证情况

### 3.4 审核记录区
路径：`project-docs/03_review_logs`

用途：
- 记录 Codex 的审核结论
- 记录通过项、问题项、风险等级

### 3.5 验证区
路径：`project-docs/04_validation`

用途：
- 保留编译、运行、ADB、盒子、截图、日志等证据

### 3.6 索引区
路径：`project-docs/05_index`

用途：
- 统一查看当前所有任务状态

## 4. 状态机基线
任务状态必须严格限定为：
- DRAFT
- IN_DEV
- DEV_DONE
- IN_REVIEW
- CHANGES_REQUIRED
- APPROVED
- VERIFIED
- CLOSED
- BLOCKED

## 5. 角色边界

### Claude
- 允许改代码
- 允许写任务卡 / 开发记录 / 验证记录 / 更新索引
- 不允许写审核结论

### Codex
- 允许写审核记录 / 更新索引
- 不允许伪装成开发已完成
- 不允许跳过代码核对直接相信开发记录

## 6. 不允许破坏的架构约束
- 一任务一编号
- 一编号对应一组文档
- 同一任务的文档编号必须一致
- 所有状态变更必须落盘
- 验证结论必须有证据
- 审核结论必须清晰，不允许模糊表述

## 7. 推荐演进方向
后续如任务量增大，可在不破坏现有目录结构前提下增加：
- `scripts/taskctl.py` 自动建档
- diff 归档脚本
- ADB 安装日志归档脚本
- 统一报告生成脚本
