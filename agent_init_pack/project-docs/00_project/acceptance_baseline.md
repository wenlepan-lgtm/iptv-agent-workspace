# acceptance_baseline.md

## 1. 目的
定义统一的验收基线，避免每次任务对“完成”的理解不同。

## 2. 通用验收要求
任一任务要进入“完成”状态，至少满足：

- 有任务卡
- 有开发记录
- 有审核记录
- 任务状态已更新
- 相关文件改动与任务目标一致
- 没有明显未声明的附带修改

## 3. 开发完成（DEV_DONE）判定条件
满足以下条件可进入 `DEV_DONE`：

- Claude 已按任务要求完成本轮代码改动
- 已写开发记录
- 已列出修改文件清单
- 已说明本轮风险与未覆盖场景
- 已写明建议的下一状态

## 4. 审核通过（APPROVED）判定条件
满足以下条件可进入 `APPROVED`：

- Codex 已核对任务要求与代码实现一致性
- Codex 已核对开发记录与代码一致性
- 审核记录已落盘
- 无 P0 / BLOCKER 问题
- 无必须立即修复的高风险问题，或已明确接受该风险

## 5. 验证通过（VERIFIED）判定条件
若任务要求编译、运行或设备验证，则除审核通过外，还必须满足：

- 已有验证记录
- 已记录实际执行命令
- 已记录环境信息
- 已记录结果与证据位置
- 关键结果满足任务卡验收标准

## 6. 关闭任务（CLOSED）判定条件
满足以下条件方可关闭：

- 任务已 APPROVED
- 若任务要求验证，则已 VERIFIED
- Ala 已确认当前轮工作完成

## 7. 失败与退回规则

### 应退回 `CHANGES_REQUIRED` 的场景
- 实现与任务要求不一致
- 开发记录与代码不一致
- 存在明显未声明改动
- 缺少关键验证信息
- 回归风险未控制

### 应进入 `BLOCKED` 的场景
- 缺少关键依赖
- 环境不可用
- ADB / 设备不可访问
- 输入条件不足无法推进

## 8. 文档质量要求

### 开发记录必须包含
- 做了什么
- 没做什么
- 改了哪些文件
- 如何验证
- 验证结果
- 风险与未覆盖项

### 审核记录必须包含
- 审核结论
- 风险等级
- 通过项
- 问题项
- 是否允许进入下一步

### 验证记录必须包含
- 验证目标
- 验证环境
- 执行命令
- 结果
- 证据位置
- 结论
