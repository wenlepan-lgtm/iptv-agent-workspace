# known_issues.md

## 1. 目的
记录当前明确存在的问题、限制与协作风险，避免重复踩坑。

## 2. 当前已知协作问题

### K-001：聊天上下文不可靠
说明：
仅依赖聊天记录会造成任务目标漂移、验证状态漂移、审核标准漂移。

当前策略：
- 一律以仓库文档为准
- 必须落盘

### K-002：任务边界容易扩散
说明：
开发时容易出现“顺手改掉一堆”的情况，导致审核失焦、回归风险上升。

当前策略：
- 一任务一卡
- 非本任务范围内容不得混入

### K-003：验证结论容易口头化
说明：
“我试过了”“应该可以”这类表述无法作为工程证据。

当前策略：
- 编译、运行、ADB、设备验证必须写验证记录

### K-004：审核意见容易空泛
说明：
只说“建议优化”“可以更优雅”没有工程价值。

当前策略：
- 审核必须落到具体文件、具体问题、具体风险等级

## 3. 当前系统性限制
- 当前仅完成初始化骨架，尚未接入自动化脚本
- 当前任务编号、索引维护主要依赖人工和工具按协议执行
- 若后续任务量变大，应补 taskctl.py 自动化
