### 1.2 关键结论（与代码对应）

1. **录音 buffer overflow 共 3 次**  
   - 系统层 `RecordThread: buffer overflow` 表示应用未及时 `read()`，导致丢帧。  
   - 2ch@48k 每帧 1920 samples，当前 `bufferSizeInBytes = SAMPLES_PER_FRAME_2CH_48K * 2`（约 20ms），单线程里再做 downmix + resample + Vosk，一旦某次处理稍慢就会溢出，**首字/首词易丢或错**（如「建议」「这 也是」）。

2. **VAD 单帧即判 VOICE_END**  
   - 代码中 `isInSpeech = frameRms >= MIN_VAD_RMS`，**一帧**低于 80 就置 false，立刻打 VOICE_END。  
   - 读诗时句间/句中短暂停顿（rms 掉到 40、4.5、27.8）就触发 VOICE_END，再经 800ms 静音触发 final，导致**长句被截成多段、只出前两句**。

3. **静夜思关键字**  
   - 正确：床前、明月光、疑是、地上、霜。  
   - 错误/多余：句首「这 也是」/「这 意思」（应为静夜思），且「举头望明月」「低头思故乡」未出现在 final 中（被提前 final + 后续在 TTS 期间或溢出后）。

## 二、根因归纳（给 ChatGPT / Claude）

| 根因 | 表现 | 代码位置/机制 |
|------|------|-------------------------------|
| **录音 buffer 过小 + 单线程阻塞** | RecordThread buffer overflow，首段或句中丢帧，首字错/断句错 | `AsrController` 2ch 路径：`bufferSize2ch = SAMPLES_PER_FRAME_2CH_48K * 2`（约 20ms）；同一线程内 read → downmix → resample → 每 100ms 调 Vosk |
| **VAD 无 hangover** | 一帧低于阈值即 VOICE_END，句中短暂停顿就「断句」 | `isInSpeech = frameRms >= MIN_VAD_RMS`，无连续多帧低于阈值才判结束 |
| **静音 800ms 即 final** | 读诗句间停顿 >800ms 就出 final，长句只出前两句 | `SILENCE_DURATION_MS = 800`，`processStateMachine` 里静音超时触发 final |

## 三、解决方案（交给 Claude 实现）

### 3.1 消除 / 减轻 buffer overflow（必做）

**目标**：让录音线程有足够缓冲，避免因偶尔处理延迟就 overflow。

- **做法 1（推荐）**：增大 2ch@48k 的 `AudioRecord` 缓冲区，例如为 **80–100ms** 的数据量（当前约 20ms）：
  - 48k × 2ch × 0.1s = 9600 samples → `bufferSizeInBytes = 9600 * 2`（或至少 60ms：5760×2）。
  - 常量可定义为 `BUFFER_2CH_48K_MS = 80`，再 `samples = 48000 * 2 * BUFFER_2CH_48K_MS / 1000`，`bufferSizeInBytes = samples * 2`。
- **做法 2（可选）**：若仍 overflow，可把「送 Vosk」移到单独线程：当前线程只负责 read → downmix → resample → 放入队列，另一线程从队列取 16k mono 送 `acceptWaveForm`，避免 Vosk 阻塞 read 循环。  
  **优先做做法 1，仍溢出再考虑做法 2。**

**代码位置**：`AsrController.kt` 中 2ch@48k 初始化处，`bufferSize2ch` 的计算与 `setBufferSizeInBytes(bufferSize2ch)`。

---

### 3.2 VAD 增加 hangover（必做）

**目标**：避免句中短暂停顿（单帧掉 below 阈值）就判 VOICE_END，减少误断句。

- **逻辑**：  
  - 仅当**连续 N 帧**（如 3–5 帧）的 RMS 都低于 `MIN_VAD_RMS` 时，才把 `isInSpeech` 置为 false 并打 VOICE_END。  
  - 若某一帧低于阈值但下一帧又高于，则保持 isInSpeech=true（或重置「连续低帧」计数）。
- **参数**：可配置，如 `asr.vad.hangover.frames = 4`（约 4×100ms=400ms 静音才结束），默认 3 或 4。
- **代码位置**：2ch 与 3ch 分支中「更新 VAD 状态」处（当前 `isInSpeech = frameRms >= MIN_VAD_RMS`），以及单声道分支的 `processStateMachine` 内同类逻辑，需统一改为「连续低帧计数 + 超过 N 帧才 VOICE_END」。

---

### 3.3 长句静音时长可配置（必做）

**目标**：读诗/长句时，句间停顿不要过快触发 final。

- **当前**：`asr.vad.silence.duration.ms` 默认 800，静音 800ms 即触发 final。
- **建议**：默认改为 **1200** 或 **1500**，并在 `config.properties.template` 中注明「长句/古诗可设 1500」。
- **代码位置**：`AsrConfig` 的 `silenceDurationMs` 默认值及文档；`processStateMachine` 中已使用该配置，无需改逻辑只改默认/推荐值。

---

### 3.4 可选：StereoToMono 日志中的 max 含义

- 日志里 `StereoToMono: meanVol=xxx, max=0` 的 max 常为 0，因打日志时 `accumulatedMonoData` 可能已被 clear 或取的是错误对象。  
- 若希望 max 有参考意义：可在 clear 之前用 `accumulatedMonoData.maxOrNull() ?: 0` 存到局部变量再打日志；或改为「本帧 16k mono 的 max」。  
- 属可选项，不影响识别率，仅便于排查。

## 五、给 ChatGPT 的讨论要点

1. **Overflow 与 buffer 大小**：是否同意将 2ch@48k 的 AudioRecord buffer 提到 80–100ms？若设备内存紧张，60ms 是否可接受？
