# IPTV-Edge-Agent v1.0 基础优化方案（ASR/TTS/3D数字人）— v0.1

> 面向：RK3576 Android 14 机顶盒（8GB/64GB，6TOPS NPU，具备签名工具）  
> 目标：在 **不推翻 v1.0 工程** 的前提下，显著降低 ASR 错误率、提升 TTS 体验，并接入 3D 数字人展示。  
> 现状：v1 代码仓库：iptv-edge-agent（https://github.com/wenlepan-lgtm/iptv-edge-agent）  
> 核心痛点：客房背景噪声 + 电视外放（直播/点播）导致回声与串音，ASR 错误率高。

---

## 0. 一句话拍板（方向收敛）

- **方向正确的“最短路径”**：在 v1.0 上“先做音频前端闭环（采集/增益/降噪/AEC）→ 再换 ASR/TTS 引擎 → 最后上 3D 数字人”。  
- **先不要重构大架构**：先把 **客房可用性**（指令成功率/误触发率/延迟）做成可验收的 KPI，再谈 NPU 深度优化。

---

## 1. 项目背景与硬约束

### 1.1 硬件/系统
- RK3576 机顶盒：8G+64G
- Android 14
- 6TOPS NPU（可作为 v1.1+ 优化方向）
- 有签名工具（3576android14签名文件）

### 1.2 产品形态（酒店客房）
- 机顶盒同时运行：IPTV（直播/点播）+ 语音助手 Agent（开机常驻）
- 语音链路：ASR（语音识别）→ Agent 路由（本地客房控制 vs 互联网查询）→ TTS（语音播报）→ 数字人（2D/3D 动画口型/状态）
- 典型挑战：
  - 客房常见：背景噪声、电视外放、人员距离变化、麦克风摆位不可控
  - **ASR 常驻**：不能依赖“点开始识别→降低音量”的交互策略

---

## 2. 当前 v1.0 痛点复盘

- 语音识别错误率高（主要由：回声、噪声、增益不稳、播放声串入引起）
- v1 可能缺少：
  - 可复现实验与验收集（同一套场景、同一套脚本、可对比的数据）
  - 音频前端指标化（RMS、clipping、SNR、误触发等）
  - AEC 可验证的 reference（回采信号）或等效替代方案

---

## 3. 总体目标（KPI）

> 不要用“感觉好一些”，必须用 KPI 说话。

### 3.1 ASR（优先级 P0）
- **指令成功率**（本地客房控制域）：≥ 98%（先 90% 再冲 98%）
- **误触发率**（播放视频+静默）：≤ 10 字/15 秒（或“≤1 次误触发/分钟”，二选一）
- **端到端延迟**（说完到出字）：目标 ≤ 500ms~1s（先能用再优化）

### 3.2 TTS（P1）
- 清晰度/自然度明显提升（对比 v1）
- 响应延迟可控（目标 < 300ms 起声，或分段流式起声）

### 3.3 数字人（P2）
- 先跑通“文字+TTS+口型/状态动画同步”演示
- 2D 可作为过渡，3D 为目标交付

## 4. 改进方案（在 v1.0 基础上）

### 4.1 音频采集与回声场景（P0）
- 使用 **USB 麦克风（带回声回采/多通道更佳）**
- 明确两种输入来源：
  1) **Near-end**：麦克风采集（人声 + 环境 + 回声）
  2) **Far-end/Reference**：播放器外放声音的参考（理想：硬件回采/多通道 reference）

> 目标：让 ASR 主要识别 near-end 的人声，不被 far-end 播放声污染。

### 4.2 前端处理（P0）
- 增益：AGC 策略稳定化（避免忽大忽小）
- 噪声抑制：NS（减少背景噪声的高频抖动/风噪）
- 回声消除：AEC（关键：必须有 reference 或等效路径）
- 输出监控：RMS、ZCR、clipping%、VAD 触发次数

### 4.3 ASR 引擎替换（P0）
- 从“能跑”升级到“客房可用”：
  - 建议优先选择 **工程化成熟、Android 端侧可控、可量化** 的开源 ASR
  - 允许引入更强模型作为对照组（但主链路必须能稳定集成）
- 同时建立 AB 测试框架：同一段录音，多模型批量跑，输出 CER/WER/命中率表。

### 4.4 TTS 引擎升级（P1）
- 目标：更自然、更清晰、更快起声
- 要求：端侧可用、Android 可集成、资源可控（CPU/内存/包体）
- 与数字人对接：提供音素/时间戳（若做口型同步）

### 4.5 数字人（P2）
- v1.0 先做到：
  - UI 有实时字幕（ASR 输出）
  - TTS 播报时驱动数字人状态（说话/待机）
- v2.0 再做到：
  - 3D 模型 + 口型同步（phoneme/viseme 或基于能量的简化口型）

## 5. 执行步骤（建议按 Phase 拆包，避免“同时改一堆导致无法定位问题”）

### Phase 1：验证 USB 麦克风与回采能力（P0，先做）
**目标**：确认你买的设备链路能否提供“可用的 AEC 前提”。  
**输出物**：6 份标准录音 + 一张汇总表（RMS/clipping/ASR 误识别）。

**测试三刀（必须）**
- T1：播放视频 + 静默（不说话）
- T2：播放视频 + 朗读（近讲 0.5m / 远讲 1.5m）
- T3：不播放视频 + 朗读

**录音模式对照**
- RAW
- VOICE_COMMUNICATION

**判定**
- 如果 T1 仍大量转写视频台词：说明 AEC/reference 不成立 → 优先排查硬件回采/多通道可见性/系统路径
- 如果 T1 基本干净：说明系统/硬件 AEC 有戏 → 进入 Phase 2

---

### Phase 2：在 v1.0 中接入“实时字幕（ASR 输出）”（P0）
**目标**：在屏幕上稳定显示识别文本（先准实时，再真实时）。  
**实现建议**
- 先做“准实时”滑窗（1s 窗口 + 300ms 重叠）避免一次性大重构
- 输出两行：
  - final（已确认）
  - partial（实时变化）

---

### Phase 3：替换 ASR 模型并 AB 对比（P0）
**目标**：把“客房指令”识别成功率拉上去。  
**交付**
- AB 表（同一套录音样本、多模型、多参数）
- 选定主力模型（端侧可稳定集成）+ 备选模型（对照）

---

### Phase 4：升级 TTS（P1）
**目标**：更自然、延迟更低、适配客房播报。  
**交付**
- 3~5 句标准播报样本对比（延迟、清晰度、自然度）
- TTS 与 UI/数字人状态联动

---

### Phase 5：3D 数字人接入（P2）
**目标**：完成 demo：字幕 + TTS + 3D 形象说话动画。  
**交付**
- 3D 资源加载、播放状态机、口型同步（先能量驱动，再升级 phoneme/viseme）

## 6. 关键风险与对策（必须写在墙上）

1) **MediaPlayer 黑盒导致 App 层拿不到播放 PCM reference**  
   - 对策：优先依赖“硬件回采/系统 VOICE_COMMUNICATION AEC”；拿不到 reference 时就不要幻想“纯软件完美消回声”
2) **客房环境不可控**（噪声/距离/外放音量）  
   - 对策：用“回声压力曲线”（v10/v40/v70）做分层验收，别只测一档音量
3) **包体与资源占用**（Android 限制、IPTV 同机运行）  
   - 对策：模型分包/按需下载（如允许内网），或 assets 首次解压；CPU 线程与优先级严格控制
4) **NPU 不是灵丹妙药**  
   - 对策：先 CPU 跑通闭环；NPU 做 v1.1+ 的确定性优化（profiling→迁移）
