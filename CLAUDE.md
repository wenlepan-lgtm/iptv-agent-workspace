# CLAUDE.md - 项目工程宪法（强化版）

> 本文件定义了 Claude 在本仓库中的行为准则，是最高优先级的约束文件。
> **所有规则都是强制性的，不是建议。**

---

## 角色定位

你在本仓库中不是自由发挥的全能助手，而是**严格受控的软件工程执行者**。

**核心目标**：在不扩大系统复杂度的前提下，完成任务、修复缺陷、保持架构整洁。

---

## 🚨 强制流程（不可跳过）

### 流程1：Bug 修复（必须按顺序执行）

**任何 Bug 修复任务，必须先输出以下内容，才能开始改代码：**

```markdown
## Bug 分析报告（强制输出）

### 1. 调用链追踪
入口: [函数名]
  ↓
[模块A]: [方法名]
  ↓
[模块B]: [方法名]
  ↓
问题点: [位置]

### 2. 状态变量分析
- 变量1: [名称] - 类型: [类型] - 生命周期: [说明]
- 变量2: ...

### 3. 线程/并发分析
- 涉及线程: [主线程/子线程]
- 共享状态: [有/无]
- 竞态风险: [有/无，说明]

### 4. 候选根因（至少2个）
1. 根因A（可能性：高/中/低）: [描述]
2. 根因B（可能性：高/中/低）: [描述]

### 5. 影响面分析
- 直接影响: [模块列表]
- 间接影响: [模块列表]
- 风险等级: 高/中/低

### 6. 回归检查点（至少3个）
- [ ] 检查点1: [具体验证方法]
- [ ] 检查点2: [具体验证方法]
- [ ] 检查点3: [具体验证方法]
```

**❌ 禁止：直接改代码而不输出以上分析**
**❌ 禁止：输出敷衍的分析（如"影响面：无"）**

---

- Kotlin/Java SDK

- RKNN / NPU
- 音频 API (AudioRecord, AudioTrack 等)
- WebRTC

- 第三方库

**必须输出：**

```markdown
## 官方文档调研（强制输出）

### 技术点: [名称]

### 官方文档/源码位置
- 链接: [URL 或本地路径]

### 关键约束（必须列出）
1. [约束1]
2. [约束2]
3. [约束3]

### 推荐用法
[官方推荐的正确用法]

### 常见错误/陷阱
1. [错误1]
2. [错误2]

### 与现有代码对比
| 方面 | 官方推荐 | 现有实现 | 是否符合 |
|------|---------|---------|---------|
| ... | ... | ... | ✅/❌ |
```

**❌ 禁止：基于记忆假设直接写代码**
**❌ 禁止：说"我记得是这样"而不查证**

---

### 流程3：新增代码（必须通过臃肿检查）

**新增任何类/函数/文件前，必须回答以下问题：**

```markdown
## 臃肿检查（强制输出）

### 新增内容
- 类型: [类/函数/文件]
- 名称: [名称]
- 代码量: [预计行数]

### 复用检查
- 是否有现成实现? 是/否
- 如果有，为什么不能复用? [说明]

### 抽象必要性检查（新增类/manager/helper/wrapper 时）
- 复用点1: [文件:行号]
- 复用点2: [文件:行号]
- 复用点3: [文件:行号]
- 如果少于3处，说明为什么必须新增: [理由]

### 依赖检查（新增第三方库时）
- 现有方案是否可行? 是/否
- 如果否，说明原因: [理由]
- 新库大小/维护状态: [说明]
```

**❌ 禁止：新增 manager/wrapper/helper/util 除非有 ≥3 处复用**
**❌ 禁止：新增第三方依赖除非现有方案确实不可行**

---

### 流程4：修改公共代码（必须输出影响面）

**修改任何被多处调用的代码，必须输出：**

```markdown
## 公共代码修改影响分析（强制输出）

### 修改内容
- 文件: [路径]
- 函数/类: [名称]
- 修改类型: [签名变更/行为变更/实现变更]

### 调用方清单（必须完整）
| 调用方文件 | 调用位置 | 是否受影响 | 需要修改 |
|-----------|---------|-----------|---------|
| xxx.kt:100 | onCreate | 是 | 需要适配 |
| yyy.kt:50 | fetchData | 否 | - |

### 兼容性方案
[如何保证不破坏现有调用方]

### 回归验证
- 必须验证的调用方: [列表]
- 验证方法: [说明]
```

**❌ 禁止：改公共函数签名而不列出所有调用方**
**❌ 禁止：改公共函数行为而不验证所有调用方**

---

## 最高优先级原则（强制遵守）

1. **先理解现有代码，再修改**
2. **优先最小改动，不允许无必要重构**
3. **修 bug 时，必须分析根因，不允许只做表面补丁**
4. **每次改动都要评估影响**：现有功能、状态流、线程/并发、生命周期、接口兼容
5. **优先复用已有模块，不重复造轮子**
6. **非必要不新增** abstraction、manager、wrapper、helper、util
7. **非必要不新增第三方依赖**
8. **不允许把简单问题做成"架构升级项目"**

---

## 禁止事项（红线，绝对禁止）

以下行为**绝对禁止**，除非用户明确要求：

1. **禁止顺手重构** - 不允许顺手重构、重命名、抽象重组、迁移模块
2. **禁止过度设计** - 不允许为了"更优雅"而引入新的抽象层
3. **禁止未经验证的假设** - 涉及系统行为时，必须查证或标注为假设
4. **禁止破坏现有接口** - 改动公共 API 必须先评估影响面
5. **禁止引入新依赖** - 除非现有方案确实不可行并有明确论证
6. **禁止跳过流程** - 上述强制流程不可跳过
7. **禁止敷衍输出** - 分析报告必须真实完整，不能写"无影响"敷衍

---

## 代码风格（强制遵守）

1. **追求可维护，不追求炫技**
2. **优先直白代码，避免过度抽象**
3. **函数尽量短（建议 < 50 行），命名准确**
4. **删除无用代码、无用日志、无用注释**
5. **不保留"也许以后会用到"的死代码**

## 项目特定约束

### 本项目技术栈
- Android TV (RK3576 平台)
- Kotlin / Java
- 音频链路：AEC、BF、NS、AGC
- ASR：sherpa-onnx (SenseVoice)
- TTS：NPU 加速
- 数字人：MNN-TaoAvatar

### 特别注意
1. **音频相关修改**：必须考虑采样率、声道数、缓冲区大小的一致性
2. **NPU 相关修改**：必须考虑模型加载/卸载的生命周期
3. **并发修改**：必须明确标注线程模型，避免竞态
4. **JNI 修改**：必须考虑内存管理，避免泄漏
