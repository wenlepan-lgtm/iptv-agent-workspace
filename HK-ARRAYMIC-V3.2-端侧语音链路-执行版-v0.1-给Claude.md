## HK-ARRAYMIC-V3.2 端侧语音链路：唤醒词 + ASR 稳定性升级（执行版 v0.1）

### 目标（本阶段）

- 在**无噪声/弱噪声、近/中/远**条件下，把 APK 实时识别稳定到可用（优先解决句首丢字/错字）。  
- AEC/NS/AGC 暂不引入（避免变量爆炸），等 Phase-0 / Phase-A 通过后再做。

### 已知事实（来自现状快照）

- 麦克风输出：USB 2ch @ 48kHz/16bit，系统识别为立体声；处理链路为：  
  `2ch@48k -> mono(avg) -> 16k -> VAD -> Vosk (vosk-model-small-cn-0.22)`  
- 现象：  
  - 《静夜思》后半段比较稳定；  
  - 句首（“床前明月光 / 疑是地上霜”）高概率错；  
  - 存在 `RecordThread buffer overflow`；  
  - TTS 后已有 `post-unmute discard`（当前 1000ms）。

---

## Phase-0：纯净环境“近/侧/远”基线验收（必须先过）

### 1. 测试集（由 Ala 录制）

- `S1_steady`：0.5~1m 正对读固定文本（《静夜思》或固定口令）  
- `S2_side`：左/右侧 1m 读同样文本  
- `S3_far`：3~4m 读同样文本  

要求：每类 ≥ 10 次，录音统一使用 tinycap：`2ch@48k/16bit/10s`。

### 2. 必须新增：APK 内「离线回放模式」

**目的**：把“模型/算法问题”与“实时采集丢帧问题”解耦。

**实现建议（给 Claude）**：

- 在 APK 增加 Debug 开关或测试模式：  
  - 从 `/sdcard/*.wav`（S1/S2/S3）读取 PCM（48k/2ch）；  
  - 走**完全相同**的 `downmix + resample + VAD + Vosk` 链路；  
  - 不通过 `AudioRecord`，而是直接从文件读数据模拟实时流。

- 输出日志（至少包含）：  
  - 每次 final 的文本；  
  - 整段处理耗时；  
  - VAD 触发点（VOICE_START / VOICE_END 时间戳、帧号）；  
  - 累计溢出/丢帧计数（如模拟情况下应为 0）。

### 3. Phase-0 验收标准

- **离线回放模式**：  
  - S1/S2/S3 文本正确率 ≥ 95%（或项目要求 100%）；  
- 判断逻辑：  
  - 若离线回放 OK 但实时采集不 OK → 进入 **Phase-0.1（实时链路修复）**；  
  - 若离线回放也不 OK → 优先检查/调整：  
    - VAD 阈值 / hangover、  
    - 重采样实现、  
    - PCM 拼接/帧边界；  
    必要时考虑**替换 ASR 引擎**（Phase-B）。

---

## Phase-0.1：实时采集链路修复（只动“丢帧 / 起步”）

聚焦：**句首丢字/错字 = 起步阶段有效语音帧缺失** 概率过高。

### 建议优化项（按优先级执行）

1. **录音 buffer 从 120ms -> 160~200ms 做 A/B**  
   - 调整 `BUFFER_2CH_48K_MS`：120 → 160 / 200；  
   - 分别记录 `RecordThread buffer overflow` 次数与分布，比较前后差异。

2. **AudioRecord 线程与 Vosk decode 解耦**  
   - 录音线程只负责：`read -> downmix -> resample -> 写入无界/有界队列`；  
   - 解码线程从队列中取 16k/mono 帧喂给 Vosk；  
   - 确保采集线程**永不被 Vosk 阻塞**，从根本降低 overflow。

3. **调整 post-unmute discard 策略**  
   - 当前为 1000ms 硬丢弃；  
   - 后续可缩短或取消，改为依赖 Phase-A 的 ring-buffer 补偿机制：  
     - 用 ring-buffer 标记 TTS 提示音区间「不可送 ASR」，  
     - 其余时间段则保留前滚缓存。

### Phase-0.1 验收

- 使用与 Phase-0 相同的 S1/S2/S3 测试集：  
  - **实时采集**的识别正确率应接近于「离线回放模式」；  
  - overflow 明显减少，不再集中出现在句首关键帧。

## Phase-A：唤醒词（“小聚小聚”） + ring-buffer 预滚动（解决句首脆弱段）

### 1. 为什么要先做唤醒词 + 预滚动

- 当前「句首」恰好落在：  
  - VAD 起步不稳定、  
  - 线程/GC 可能抖动、  
  - TTS/mute/unmute 切换的时间窗。  
- 唤醒词把**业务命令的真正开始时间**推迟到系统完全稳定之后；  
- ring-buffer 则可以把「刚起音的那一口气」补齐，即使有前面少量丢帧，也能通过预滚动覆盖。

### 2. 交互设计（建议）

- 开场白 TTS：仅用于介绍（可稍长，不进入命令模式）。  
- 真正开始说话提示：播放 `beep.wav`（50~120ms），**不要用 TTS 当提示音**。  
- 用户流程：  
  1. 听到系统准备好（如开场白），  
  2. 说：「小聚小聚」 → KWS 命中，  
  3. 播 beep，  
  4. 进入命令识别窗口（例如 4 秒），在窗口内说业务命令。

### 3. 工程实现关键点

1. **Ring-buffer**  
   - 在 48k/2ch 或 16k/mono 阶段维护环形缓冲（至少 1.0s 历史）。  

2. **KWS 命中后处理**  
   - 播放 beep：  
     - 同时将「beep 开始前 300ms ~ beep 结束后 300ms」标记为**不可送 ASR**的区间（防止把提示音本身当命令）。  
   - 启动命令 ASR 时：  
     - 从 ring-buffer 中取回 beep 前 300~600ms 语音，作为**命令句子的前滚音频**一并喂入。  
     - 之后继续实时送入后续采样。

3. **VAD 策略（KWS 之后）**  
   - 在命令窗口的起始 `T0` 后，可以：  
     - **临时降低起步 VAD 阈值**，或  
     - 直接强制 `isInSpeech=true` 一小段时间（例如 500ms），  
   - 目的是减少「刚开口时被判为静音」的概率。

### 4. KWS 选型（端侧）

- **优先**：`sherpa-onnx` keyword spotting（有 Android 示例，可 JNI 集成）。  
- **备选**：Picovoice Porcupine（成熟，但需确认 AccessKey/授权策略）。  
- **不建议**：用 Vosk 的「热词」硬凑 KWS（延迟和误触控制较差）。

### 5. Phase-A 验收

- **误唤醒率**：在安静环境下运行 10 分钟，误唤醒次数 ≤ 指定阈值（例如 ≤1 次）。  
- **唤醒后第一句命令识别**：  
  - 与 Phase-0 对齐，同样的 S1/S2/S3 条件下，  
  - 句首关键字（例如《静夜思》中的“床前明月光”）识别稳定性显著提升，不再长期偏为「朕叶子/行情呢着/歌房钱」等。

---

## Phase-B：是否替换 Vosk（仅在 Phase-0/0.1 仍不达标时）

### 判断规则

- 若 **离线回放** 模式下，Vosk 仍然在句首表现糟糕（静夜思首句长期错），且经排查 VAD/拼接等逻辑无明显 bug →  
  - 可以认为主要瓶颈在于 Vosk 中文声学/语言模型。  
  - 启动 ASR 引擎替换评估。

### 替换方向

- 优先评估：  
  - `sherpa-onnx` ASR（与 KWS 属同一生态，便于端侧统一部署）。  
- 其他端侧 ASR：  
  - 必须满足：有可落地的 **Android SDK / JNI 接口**，且能承载 16k/mono 实时识别。

---

## Phase-C：AEC / NS / AGC（最后做）

### 前提条件

- Phase-0 / Phase-0.1 / Phase-A 已通过：  
  - 在「安静/弱噪近中远」场景下，  
  - APK 实时识别的稳定性已接近离线回放基线，  
  - 句首问题主要已由 KWS + ring-buffer + 链路修复解掉。

### 引入策略

- 再逐项引入：  
  - AEC（回声消除）、  
  - BF（波束成形）、  
  - NS（降噪）、  
  - AGC（自动增益）。  
- 每引入一项，都要求有**明确的 A/B 对比实验**与回退方案：  
  - 单独打开某一模块，比较前后在 S1/S2/S3 和典型实际场景下的识别率变化；  
  - 若效果不达标或引入不稳定性，可一键关闭该模块回退到上一阶段。
