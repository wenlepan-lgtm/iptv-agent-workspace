# IPTV Edge Agent v1 语音链路优化/重构方案（讨论稿）
**面向：RK3576 Android TV 端侧语音助手（客房控制 vs 互联网指令分流）**  
**版本：v0.1（2026-01-29）**  
**仓库：iptv-edge-agent v1**：https://github.com/wenlepan-lgtm/iptv-edge-agent

---

## 1. 当前现状与核心痛点（结论先行）
### 1.1 你遇到的“高错误率”本质上是两类问题叠加
- **音频前端问题（占大头）**：回声/背景噪声/增益失控/削波(clipping) → 直接把 ASR 喂瞎。
- **ASR 后端问题（次要但重要）**：模型/解码/热词/语言模型约束不足 → 在酒店场景（电视外放 + 远场）进一步恶化。

### 1.2 KPI 要校准：别盯“开放式 WER 1%”
- **“开放式口语转写 WER≈1%”**：酒店远场 + 回声环境下，端侧本地 ASR 基本不现实（尤其中文、混响、多人说话）。
- **合理 KPI（业务向）**：
  - **客房控制域**：Command Success Rate（指令成功率）→ 目标 98%~99%（通过约束解码/热词/确认机制可达）。
  - **互联网域**：可用性优先（允许失败后兜底：复述确认/改问法/联网/降级）。

## 2. 你提出的两个验证点：最小闭环验收口径（必须可观测）
你提出的两点是对的，但需要更“工程化”的验收口径，否则会陷入无穷试错。

### 2.1 验证 USB 麦克风（带回声参考通道）是否“真的有用”
**目标**：确认 6 通道（4 MIC + 2 Reference 回采）的数据质量满足 AEC + ASR 的工程要求。

**必须验证的 4 件事（全部可量化）**：
1) **Reference 通道是否干净**
   - 只包含系统播放声（TV 外放），尽量不混入人声/环境声。
2) **MIC 通道增益是否可控且不削波**
   - clipping 一旦出现，ASR 会“稳定发疯”，后面换模型也救不了。
3) **AEC 是否有效**
   - 用回声测试音（TTS/白噪/扫频）比对 AEC 前后能量与残留回声。
4) **NS/AGC/VAD 后人声是否被破坏**
   - 很多“降噪”会把辅音/尾音抹掉，反而更难识别。

**验收建议（建议阈值，可调整）**
- 回声场景：AEC 后相对可感知地回声降低（主观）+ 指标明显改善（客观）。
- 近讲(0.5m)与远讲(1~2m)都要测。
- 不满足：优先修音频链路，不要急着换模型。

---

### 2.2 更换更优秀的开源 ASR：先选“可交付模型”，再冲“实验型模型”
建议按“交付风险”从低到高排序：

**Tier-1（交付优先）**
- **sherpa-onnx + SenseVoiceSmall（int8，流式）**  
  优势：跨平台路径清晰，Android/JNI 落地更稳；流式延迟更可控；工程资料多。  
  适合：v1 交付主力。

**Tier-2（性能/精度进一步探索）**
- sherpa-onnx 生态内其他 streaming 模型（按中文/延迟/体积选）。
  适合：v1 稳定后做替换/AB。

**Tier-3（实验型：LLM-based ASR）**
- **Fun-ASR-Nano-2512 + INT8 + CPU**  
  风险点：导出/端侧推理/生态适配不确定性更高，Python 跑通 ≠ Android 稳定交付。  
  适合：并行探索“精度上限”，不建议作为 v1 主路径。

## 3. 你问的两个问题：直接结论
### Q1：Fun-ASR-Nano-2512 + INT8 + CPU 推理，Claude+GLM4.7（Ubuntu/Mac）能否帮你处理？
**能，但仅适合“离线实验闭环”，不适合作为 v1 主交付路线。**

Claude+GLM4.7 能做：
- 拉起推理 demo、跑基准（RTF/延迟/CPU/内存）
- 做 INT8 量化尝试（PTQ/QDQ 路线）
- 设计 AB 测试脚本与统计口径（CER/WER、指令成功率）

但你最终要交付在 Android（JNI/C++，长时间运行，低延迟、低占用）：
- 生态适配成本可能高
- 维护成本高
- 一旦与音频前端/多线程/电源管理耦合，问题会很“脏”

**建议定位**：Fun-ASR-Nano-2512 = “候选验证项/精度上限探索”，别压在 v1 交付主线。

---

### Q2：既然有 NPU，CPU 还要留给 IPTV/直播/点播，还有更好的方案吗？
**有：把“可交付链路”先跑通，再逐步吃 NPU。**

**推荐 v1 主路线（工程稳、可交付）**
- 音频前端：**WebRTC APM（AEC/NS/AGC/VAD）**
- ASR：**sherpa-onnx（流式）+ SenseVoiceSmall int8**
- 分流策略：客房控制域强约束（热词/语法/二次确认）；互联网域可弱约束（必要时联网/降级）

**关于 NPU**
- 原则：**必须以“实测利用率 + 端到端延迟”验收**，不要靠想象。
- 实操：先验证 Android 侧是否能稳定获取 6 通道，再看是否能把 ASR 的 encoder 部分/部分算子走 rknn/rknpu2。
- 风险：NPU 生态经常出现“看似支持但实际没吃到”的情况（CPU 飙高、NPU 低占用），必须用 profiling 证据说话。

## 4. 结构性改造建议：把“客房控制”从“开放式 ASR”里剥离
你说要把“客房控制指令”和“互联网指令”分开 —— 这是关键。

### 4.1 客房控制域（强约束 → 追求 99% 任务成功率）
目标不是“完美转写”，而是“稳定执行”：
- 小模型流式 ASR + **热词/语法约束**
- 低置信度触发 **二次确认**
- 允许把识别目标变为：**Command Success Rate**（而不是 WER）

### 4.2 互联网域（弱约束 → 追求可用性/体验）
- 允许更大模型（本地更强/或联网）+ 语义纠错
- 但要限频、可降级，避免抢占 IPTV/解码/渲染 CPU

## 5. 最小可执行“验证任务包”（你下一步就按这个打）
> 核心要求：每一步都有验收指标，有失败分支（回退/修复），避免“玄学调参”。

### Task-1：Android 侧验证 USB 6 通道采集
- 验证 AudioRecord/AAudio 能否取到完整 6 通道
- 输出：原始多通道 WAV（含 time-aligned 的 reference）

**验收**
- 通道数正确
- reference 通道播放声清晰
- mic 通道无严重削波/爆音

### Task-2：AEC/NS/AGC 前后对比（离线）
- 同一段录音：
  - raw
  - AEC 后
  - AEC+NS 后
  - AEC+NS+AGC 后

**验收**
- 回声显著下降（主观 + 客观）
- 人声清晰度不下降（特别是尾音/轻辅音）

### Task-3：ASR AB 测试（同一批样本）
- SenseVoiceSmall int8（流式/离线都可）
- Fun-ASR-Nano-2512（离线）
- 输出：CER/WER + 指令成功率（按指令集）

**验收**
- 客房控制域：成功率明显提升（优先）
- 互联网域：可用性提升（次要）

### Task-4：CPU 预算表（必须做）
- IPTV/直播解码
- 点播
- UI/数字人
- AEC/NS
- ASR

**验收**
- 明确“在哪个场景会打满 CPU”
- 明确“何时触发降级策略”（如停数字人、切低码率、暂停联网ASR等）

## 6. 决策建议（像开会拍板）
**如果目标是 4~8 周内交付稳定 v1：**
- 主线：WebRTC APM + sherpa-onnx + SenseVoiceSmall int8
- Fun-ASR-Nano-2512：并行探索，不进入主线
- KPI：客房控制域“指令成功率”优先，不再死磕开放式 WER

**如果目标是冲精度上限（愿意承担不确定性）：**
- Fun-ASR-Nano-2512 可以做，但你需要为“端侧适配/导出/性能问题”预留成本与时间。
