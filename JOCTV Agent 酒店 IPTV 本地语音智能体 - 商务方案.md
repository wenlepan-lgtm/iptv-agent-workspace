# JOCTV Agent 酒店 IPTV 本地语音智能体

# 商务销售方案 v1.0

---

## 一、产品概述

### 1.1 一句话定义

**JOCTV Agent 是部署在酒店客房电视终端上的本地 AI 语音智能体**，住客对着电视说话就能控制客房设备、呼叫酒店服务、查询本地与实时信息，系统由 3D 数字人形象进行语音与视觉交互。

### 1.2 产品形态

| 项目 | 说明 |
|------|------|
| 运行位置 | 客房电视终端（机顶盒），非云端、非外挂音箱 |
| 交互方式 | 远场语音唤醒 + 语音对话 + 3D 数字人视觉反馈 |
| 核心能力 | 语音识别、意图理解、服务路由、语音合成、数字人驱动 |
| 部署模式 | 终端本地部署为主，云端增强为辅 |
| 目标场景 | 酒店客房 IPTV 系统 |

### 1.3 核心差异化

市面上大多数酒店语音方案是"云语音 + 智能音箱"或"云语音 + 外挂设备"。JOCTV Agent 的根本不同在于：

> **语音处理在客房本地完成，不需要把住客语音上传到云端。**

这意味着：
- **隐私安全**：原始语音不出房间，天然满足酒店业对住客隐私的保护要求
- **稳定可靠**：断网、弱网场景下核心功能（客房控制、本地服务）仍然可用
- **响应更快**：端侧处理，减少云链路延迟，客控响应更即时
- **成本可控**：端侧优先降低云端 API 调用量，规模化部署时运营成本更低

## 二、市场痛点与机会

### 2.1 酒店行业现状痛点

| 痛点 | 具体表现 |
|------|---------|
| **住客体验单一** | 客房电视仍以被动内容播放为主，缺乏主动交互能力，住客期望智能化体验 |
| **语音方案隐私风险** | 云端语音方案需上传原始语音，存在住客隐私泄露与合规风险 |
| **外挂设备体验差** | 智能音箱方案需额外设备，占桌面空间、需充电、体验割裂 |
| **网络依赖高** | 纯云方案在网络波动或中断时完全不可用，影响住客使用 |
| **运营能力弱** | 现有方案大多"黑盒"运行，酒店无法自定义话术、知识、服务策略 |
| **集成成本高** | 语音能力与 IPTV、RCU 客控系统各自独立，需多次集成对接 |

### 2.2 市场机会

- **酒店智能化升级**：中高端酒店集团正全面推动客房智能化改造，语音交互是核心入口之一
- **政策与合规驱动**：住客隐私保护法规趋严，本地化处理方案更具合规优势
- **端侧 AI 技术成熟**：NPU 算力提升使端侧语音识别与合成成为可行方案
- **数字人交互趋势**：3D 数字人从概念走向落地，酒店场景是天然应用场景
- **IPTV 渗透率高**：酒店客房 IPTV 覆盖率持续提升，是天然的交互终端

### 2.3 目标客户画像

| 客户类型 | 特征 | 核心诉求 |
|----------|------|---------|
| **连锁酒店集团** | 数百~数千家门店，统一品牌标准 | 标准化部署 + 门店差异化运营 + 集团级管控 |
| **高端/精品酒店** | 注重住客体验与品牌差异化 | 科技感体验 + 品牌定制数字人形象 |
| **文旅酒店/度假村** | 场景丰富，服务需求多元 | 本地知识问答 + 周边推荐 + 多语言支持 |
| **酒店 IPTV 运营商** | 为酒店提供 IPTV 解决方案 | 产品能力集成 + 增值服务 + 降本增效 |
| **酒店智能化集成商** | 承接酒店弱电/智能化项目 | 标准化接口 + 快速部署 + 可靠交付 |

## 四、产品功能模块

### 4.1 核心功能清单

| 功能模块 | 能力说明 | 住客体验 |
|----------|---------|---------|
| **语音唤醒** | 远场唤醒词触发（如"小智小智"） | 无需遥控器，自然语音唤起 |
| **语音识别（ASR）** | 本地 NPU 加速，支持中英日韩粤 | 说话即识别，实时显示识别文字 |
| **意图理解与路由** | 50+ 意图规则，智能分类到对应服务 | 精准理解需求，快速执行 |
| **客房控制** | 灯光、空调、窗帘、电视控制 | "把空调调到 26 度"、"关灯"一句话搞定 |
| **酒店服务呼叫** | 前台、客房服务、机器人配送 | "叫前台"、"送一瓶水"直接触发 |
| **本地知识问答** | 早餐时间、WiFi 密码、酒店设施、周边景点 | "早餐几点开始？"、"附近有什么好玩的？" |
| **实时信息查询** | 天气、新闻、时间、汇率、股票 | "明天天气怎么样？" 获取真实数据 |
| **通用问答** | 接入云端大模型，回答开放性问题 | "帮我讲个笑话"、"李白是哪个朝代的" |
| **语音合成（TTS）** | 本地语音合成，中英文双链路 | 自然语音回复，非机械合成音 |
| **3D 数字人** | 口型同步、表情联动、状态动画 | 有形象感的对话体验 |
| **合规与安全** | 敏感词拦截、越权拒答、内容审计 | 安全可控，不回答不适当内容 |

## 五、竞争优势

### 5.1 与主流方案对比

| 对比维度 | 智能音箱方案 | 纯云语音方案 | **JOCTV Agent** |
|----------|-------------|-------------|-----------------|
| 部署位置 | 外挂设备 | 云端 | **电视终端本地** |
| 隐私安全 | 弱（语音上云） | 中（语音上云） | **强（本地处理）** |
| 断网可用性 | 低 | 低 | **高（核心能力离线）** |
| 响应延迟 | 中（含云链路） | 中~高 | **低（端侧直处理）** |
| 酒店可运营性 | 弱（黑盒） | 中 | **强（可配置可编辑）** |
| IPTV 集成度 | 低（外挂） | 中（需对接） | **高（原生集成）** |
| RCU 客控联动 | 弱 | 需二次开发 | **标准化直连** |
| 交互形态 | 语音播报 | 语音播报 | **语音 + 3D 数字人** |
| 多语言支持 | 视品牌 | 视供应商 | **中英日韩粤五语** |
| 长期运营成本 | 中（持续云费） | 高（云调用费） | **低（端侧优先降云费）** |

### 5.2 核心技术壁垒

1. **端侧 AI 全链路**：ASR + TTS + 意图路由全部在终端完成，不依赖云端算力
2. **NPU 加速**：基于 RK3576 NPU 的模型推理加速，端侧性能达到实用水平
3. **3D 数字人端侧渲染**：不依赖云渲染，口型与语音实时同步
4. **音频工程深度优化**：回声消除、噪声抑制、背景音门控等客房场景专项优化
5. **可运营架构**：意图路由、知识库、话术、策略均为配置驱动，酒店可自主运营

## 七、部署与实施

### 7.1 部署架构

```
酒店集团云平台
    ↕（配置下发、日志汇聚、策略更新）
门店局域网
    ↕
客房终端（机顶盒）
  ├── JOCTV Agent（常驻进程）
  ├── IPTV 系统
  └── RCU 客控接口
```

### 7.2 实施流程

| 阶段 | 周期 | 工作内容 | 交付物 |
|------|------|---------|--------|
| **需求对接** | 1~2 周 | 客户场景梳理、定制需求确认、硬件环境评估 | 需求确认书、技术方案书 |
| **系统适配** | 2~4 周 | IPTV 系统对接、RCU 协议适配、PMS/服务系统对接 | 对接接口文档、联调记录 |
| **内容配置** | 1~2 周 | 酒店知识库录入、意图规则配置、话术定制、数字人形象适配 | 配置清单、验收测试用例 |
| **现场部署** | 1~2 周 | 试点楼层/门店安装部署、现场调试、运维培训 | 部署报告、运维手册 |
| **试运行** | 2~4 周 | 试运行监测、问题修复、体验优化 | 运行报告、优化记录 |
| **正式上线** | - | 全面推广、持续运维支持 | 上线确认书 |

### 7.3 实施保障

- **专项团队**：技术对接、现场部署、运维支持专人负责
- **分阶段交付**：支持试点先行、分批上线，降低一次性切换风险
- **降级机制**：网络异常时自动降级到本地能力，不影响基础功能
- **回滚预案**：每阶段交付均具备回滚能力

## 十、成功案例与应用前景

### 10.1 典型应用场景

**场景一：商务酒店**

> 入住商务酒店的旅客回到房间，说"小智小智，把空调调到 24 度，窗帘关上，明天早上 7 点叫醒我"。电视上的数字人确认执行，同时推送明日天气。旅客无需翻找遥控器、无需研究空调面板，一句话完成全部操作。

**场景二：度假酒店**

> 一家三口入住度假酒店，孩子问"电视里的姐姐，这里有游泳池吗？几点开放？"。数字人回答游泳池的位置、开放时间、儿童区说明，并推荐当天的亲子活动。

**场景三：国际旅客**

> 外国旅客用英语说"What's the WiFi password?"，系统识别英文意图并回复 WiFi 信息。旅客询问附近餐厅推荐，系统基于酒店配置的周边信息用英文回答。

**场景四：酒店集团运营**
